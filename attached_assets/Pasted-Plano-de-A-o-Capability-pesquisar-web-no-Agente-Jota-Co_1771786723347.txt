Plano de Ação: Capability pesquisar_web no Agente Jota
Contexto dos Benchmarkings (Consolidado)
Antes do plano, os achados finais dos concorrentes que informam cada decisão:

Manus AI 1.6: Arquitetura "Wide Research" com até 100 agentes paralelos. O diferencial não é o modelo de linguagem, mas a infraestrutura de execução — sandbox Linux, browser automation, CodeAct (código Python como mecanismo de ação). Para busca: usa Playwright para navegar a web como um humano, extrai conteúdo real das páginas, compila relatórios em Word/Excel. Não tem nenhuma especialização educacional.

Kimi K2.5 Agent Swarm: Paralelização de agentes especializados. K1.5 buscava em 100+ sites em tempo real. K2.5 adicionou visão multimodal. O mecanismo de tool use acontece dentro do chain-of-thought reasoning — a IA "pensa em voz alta" e decide quando chamar ferramentas. Completamente agnóstico de domínio.

Genspark AI: Mixture-of-Agents com 9 LLMs especializados + 80 ferramentas integradas. O card de "Sparkpages" mostra cada etapa de busca em tempo real com sumário dinâmico por fonte. É o sistema visualmente mais transparente dos 9 — usuário vê exatamente o que cada agente encontrou e por que. Sem foco educacional.

O gap que vamos preencher: Nenhum desses sistemas sabe que existe BNCC, Nova Escola, Scielo.br, ou EducaCAPES. Todos tratam um blog de receitas com o mesmo peso de um artigo peer-reviewed do Scielo. Para um professor brasileiro, isso é inaceitável — e é exatamente o que vamos resolver.

MACRO 1 — Infraestrutura de Busca Backend
Objetivo: Criar o motor de busca educacional como endpoint de API. Assim que concluído, já é possível testar a busca diretamente e verificar se os resultados são pedagogicamente superiores às ferramentas genéricas.

Por que primeiro: Toda a capability depende desta base. Se o backend não funcionar, nada acima funciona. E ao fazer isso primeiro, temos um resultado concreto e testável imediatamente — pode-se fazer um curl e ver resultados reais.

Micro 1.1 — Verificação dos Secrets de Ambiente
Como está agora: Não se sabe quais chaves de API de busca estão disponíveis no ambiente. O código da capability precisará ser diferente dependendo do provider disponível (Tavily vs. fallback gratuito).

Como deve ficar depois: Verificação completa de TAVILY_API_KEY, SERPER_API_KEY, e BRAVE_SEARCH_API_KEY nos secrets de ambiente. Decisão documentada de qual provider usar. Se Tavily existir: ele será o primário. Se não existir nenhuma chave: implementar com DuckDuckGo HTML + Jina Reader (100% gratuito, sem registro).

Impacto: Determina toda a implementação subsequente do backend.

Micro 1.2 — Criar arquivo api/search.js
Como está agora: Não existe nenhum arquivo api/search.js. Não há rota de busca web no backend. O api/server.js tem rotas para email, perfis, atividades, calendário — mas zero capacidade de buscar na internet.

Como deve ficar depois: Um arquivo api/search.js existe com:

Rota POST /api/search/web aceitando { query, queries?, search_depth?, max_results? }
A whitelist educacional embutida com 15+ domínios brasileiros em ordem de prioridade
Suporte a múltiplas queries paralelas (usando Promise.all)
Retorno estruturado: { results: [...], query, total_found, provider, duration_ms }
Cada resultado no array results terá: { title, url, content, snippet, score, domain_tier, source_type, published_date }.

Micro 1.3 — Provider de Busca com DuckDuckGo + Jina Reader
Como está agora: Sem Tavily, não há forma gratuita de buscar. DuckDuckGo tem uma interface HTML não oficial que pode ser raspada com cheerio. Jina Reader (r.jina.ai/{url}) converte qualquer URL em Markdown limpo, sem custo e sem autenticação.

Como deve ficar depois: O sistema consegue buscar em 2-3 queries paralelas via DuckDuckGo HTML, extrair título, URL e snippet de cada resultado, e para as top-3 URLs com maior score educacional, usar Jina Reader para extrair o conteúdo completo da página. O resultado de uma busca por "fotossíntese 7º ano BNCC" retorna resultados reais da Nova Escola, MEC, e Scielo — não de blogs aleatórios.

Nota importante: Se Tavily estiver disponível, substitui o DuckDuckGo inteiramente, mantendo o Jina Reader apenas para extração de conteúdo completo. Tavily tem include_domains nativo, o que elimina a necessidade de whitelist manual na busca.

Micro 1.4 — Motor de Re-ranking Educacional
Como está agora: Sem re-ranking. Resultados de busca genérica aparecem em ordem de relevância genérica do mecanismo de busca. Um blog de um professor desconhecido poderia aparecer antes de um documento oficial do MEC.

Como deve ficar depois: Função calculateEducationalScore(result) que aplica a fórmula consolidada dos benchmarkings:

score_final =
  0.40 × relevância_semântica   (keyword overlap com query original)
  0.25 × autoridade_domínio     (gov.br=100, edu.br=90, org.br curado=85, .br geral=50, outros=20)
  0.15 × recência               (conteúdo atual=1.0, 1-2 anos=0.8, 3+ anos=0.5)
  0.10 × tipo_conteúdo          (PDF oficial=1.3x, .gov=1.4x, artigo acadêmico=1.2x, blog=0.7x)
  0.10 × boost_educacional_br   (novaescola, mec, bncc, scielo = +0.4 flat)
Resultado: um artigo da Nova Escola sobre metodologias ativas SEMPRE aparece antes de um artigo genérico de autoajuda que casualmente menciona "educação". Isso é o que nos separa das plataformas genéricas.

Micro 1.5 — Registrar Rota no api/server.js
Como está agora: api/server.js não importa nem monta nenhuma rota de busca web. Qualquer chamada a /api/search/web retornaria 404.

Como deve ficar depois: api/server.js tem import searchRoutes from './search.js' e app.use('/api/search', searchRoutes). Um curl -X POST http://localhost:3001/api/search/web -H "Content-Type: application/json" -d '{"query":"plano de aula fotossíntese 7o ano BNCC"}' retorna JSON com resultados reais ranqueados.

Resultado tangível do MACRO 1: A API de busca educacional existe, funciona, e já produz resultados superiores a qualquer ferramenta genérica. Pode ser testada independentemente de qualquer outra parte do sistema.

Micro 2.2 — Query Planning Educacional com LLM Interno
Como está agora: Nenhuma capability existente faz query planning. Todas recebem parâmetros diretos e os usam literalmente. O professor precisaria saber exatamente o que pesquisar.

Como deve ficar depois: Antes de qualquer busca, pesquisarWebV2 faz uma chamada ao Groq (modelo llama-3.1-8b-instant — rápido e barato) com um prompt de sistema específico para gerar 2-4 queries educacionais otimizadas.

O prompt de sistema do query planner será algo como:

"Você é um especialista em busca educacional brasileira. Dado o pedido de um professor, gere 2-4 queries de busca otimizadas cobrindo aspectos diferentes: curricular (com BNCC), pedagógico (plano de aula/atividade), prático (recursos baixo custo), acadêmico (scielo, capes). Retorne JSON com array de queries."

Para "quero criar uma aula sobre frações para o 5º ano usando jogos", o planner gera:

[
  "frações 5º ano Matemática BNCC EF05MA07 habilidades",
  "plano de aula frações ensino fundamental jogos pedagógicos",
  "atividades lúdicas frações anos iniciais material concreto baixo custo",
  "jogos matemáticos frações 5 ano site:novaescola.org.br OR site:mec.gov.br"
]
Isso é o que separa nossa pesquisa de uma busca literal — inteligência pedagógica na formulação das queries.

Micro 2.5 — Lógica de Gap Analysis e Critério de Parada
Como está agora: Nenhuma capability tem lógica iterativa. Todas executam uma vez e retornam.

Como deve ficar depois: pesquisarWebV2 implementa a lógica de convergência inspirada no Manus:

function analyzeGaps(results: SearchResult[], focus_type: string): GapAnalysis {
  return {
    has_curricular: results.some(r => r.content.includes('BNCC') || r.domain.includes('mec.gov')),
    has_pedagogical: results.some(r => r.source_type === 'plano_aula' || r.source_type === 'atividade'),
    has_practical: results.some(r => r.content.includes('material') || r.content.includes('baixo custo')),
    has_academic: results.some(r => r.domain.includes('scielo') || r.domain.includes('capes')),
    coverage_score: /* weighted score */,
  };
}
// Para quando coverage_score >= 0.80 ou quando rounds >= max_rounds (default: 2)
// Nunca mais de 3 rounds — máximo de 15s total
A segunda rodada só acontece se a análise de gaps identificar lacunas reais. Isso evita buscas redundantes e mantém o tempo total aceitável (< 10 segundos para pesquisa avançada).

Resultado tangível do MACRO 2: A capability existe, funciona localmente, produz debug log multi-rodada visível no card, e gera prompt_context rico. Pode ser testada chamando a função diretamente, mesmo sem estar registrada no executor.

Micro 4.2 — Injeção de Contexto Web no Executor para Capabilities Downstream
Como está agora: O executor tem ensureBnccExecution() que injeta contexto BNCC automaticamente antes de gerar_conteudo_atividades. Não existe equivalente para contexto web. Mesmo que pesquisar_web rode antes de criar_arquivo, o criar_arquivo não sabe que os resultados existem.

Como deve ficar depois: No executor.ts, antes de executar criar_arquivo, gerar_conteudo_atividades, adicionar:

// Extrai contexto de pesquisa web do capabilityResultsMap
const webSearchResult = this.capabilityResultsMap.get('pesquisar_web');
if (webSearchResult?.data?.prompt_context) {
  enrichedParams.web_search_context = webSearchResult.data.prompt_context;
  enrichedParams.web_sources_found = webSearchResult.data.results?.length || 0;
  // Log no debug para o professor ver
  debug_log.push({
    type: 'info',
    narrative: `🌐 Contexto web injetado: ${webSearchResult.data.results?.length} fontes educacionais disponíveis para enriquecer o conteúdo`
  });
}
As capabilities criar_arquivo e gerar_conteudo_atividades já recebem enrichedParams, então web_search_context fica disponível automaticamente no input.context dessas capabilities. Seus prompts LLM internos precisarão de uma instrução: "Se web_search_context estiver disponível, USE as fontes listadas e cite-as no resultado."

NÃO implementar auto-execução forçada — diferente do ensureBnccExecution() que é obrigatório, pesquisar_web é enriquecimento opcional. Se o planner não a incluiu, é porque não foi necessária.

▐ BLOCO 1 — ARQUITETURA GERAL DO SISTEMA DE BUSCA
1.1. Passo a passo completo (do momento que decido buscar até a resposta ao usuário)

Raciocínio inicial (meu CoT interno): Leio a query do usuário + histórico da conversa. Avalio se meu conhecimento pré-treino (cortado em data variável, mas sempre suplementado por tools) é suficiente, atual ou verificável. Se a query contém indicadores temporais ("2026", "agora", "últimas"), fatos específicos pós-corte, contradições potenciais ou necessidade de fontes primárias → decido buscar.
Planejamento de queries (veja 1.5 e Bloco 5): Gero 1-n queries otimizadas, decido parallel vs sequential.
Output da function call: Imprimo exatamente o formato XML definido no meu system prompt. Exemplo real que eu já usei nesta conversa:

XMLrun web_search with query is Grok xAI web search architecture num_results is 15
run browse_page with url is https://docs.x.ai/developers/tools/web-search instructions is Extraia schema completo, parâmetros, retornos...
(Posso colocar 2-5 calls em paralelo no mesmo output.)

Execução server-side (xAI infrastructure): O sistema executa os tools de forma autônoma (agentic server-side tool calling, introduzido em 2025). Para web_search: consulta motor real-time + browsing automático de top results. Retorna resultados como texto estruturado na próxima mensagem do sistema.
Processamento dos resultados: Nova mensagem chega com os outputs dos tools. Eu os leio, faço re-ranking mental, reconciliação, síntese, grounding.
Iteração (se necessário para deep research): Se info insuficiente, repito o ciclo (novo CoT + novas calls). Critério de parada: info suficiente + consistente ou limite prático de tokens/iterações.
Resposta final: Gero texto com citações inline, síntese, e entrego ao usuário. Todo o fluxo é stateless por mensagem, mas mantém contexto acumulado via histórico.

1.2. Sim, uso Tool Use / Function Calling 100% das vezes para busca.
Schema exato da ferramenta web_search (do meu prompt atual + docs.x.ai confirmados):
JSON{

  "description": "This action allows you to search the web. You can use search operators like site:reddit.com when needed.",

  }

browse_page (usado para conteúdo completo):
JSON{
  "name": "browse_page",
  "description": "Use this tool to request content from any website URL. It will fetch the page and process it via the LLM summarizer, based on instructions.",
  "parameters": {
    "type": "object",
    "properties": {
      "url": { "type": "string" },
      "instructions": { "type": "string", "description": "Custom prompt guiding the summarizer" }

  }

Retorno típico (injetado como texto na minha próxima mensagem):

web_search: lista de 10-30 itens com title, url, snippet + data aproximada.
browse_page: texto limpo + extraído conforme instructions (ex: "Extraia seções sobre BNCC e liste habilidades de Matemática 6º ano").

1.3. Decisão de buscar vs conhecimento interno
Lógica: RL-trained emergent behavior. Gatilhos explícitos no meu treinamento:

Palavras-chave temporais, eventos recentes, estatísticas 2025+, "atual", "novidade".
Queries factuais específicas ou que exigem verificação (ex: preços, notícias, leis novas).
Quando detecto possível desatualização (meu conhecimento base é pré-2025 em muitos domínios, mas tools tornam tudo real-time).

Não existe "limiar de confiança" numérico exposto a mim (é interno ao modelo). Eu simplesmente raciocino: "Meu conhecimento para BNCC é estável, mas para recursos 2026 preciso buscar". [ESPECULAÇÃO: provavelmente há um classifier interno ou token de uncertainty no reasoning model.]
1.4. Paralelização
Sim. Posso emitir múltiplas function calls no mesmo output (até 5-10 na prática). Decido com base em decomposição da query: queries independentes → parallel; dependentes → sequential + iterações.
1.5. Fase de planejamento de queries
Sim, sempre. No CoT eu faço:

Decomposição (query → sub-perguntas).
Query expansion (sinônimos, operadores site:.gov.br, after:2025).
Priorização (1 query broad + 2-3 targeted).

Exemplo interno real desta conversa: "Grok web search architecture" → paralelizei com 4 queries diferentes + 3 browses em docs.x.ai.

2.2. Índice próprio ou real-time?
Real-time + possível índice cacheado (docs.x.ai falam "real-time search" e "live search API"). Não é só índice estático: web_search + browse_page permite fetch fresco de qualquer página. Frequência: contínua para X, real-time para web. Abrangência: milhões de páginas (estimativa pública ~14M em um artigo de 2025, mas não oficial).
2.3. Fontes não indexadas publicamente?
Não. Apenas conteúdo público acessível via HTTP. Sem paywalls, login ou APIs privadas exclusivas.
2.4. Ranking/hierarquia
Eu faço re-ranking manual no raciocínio: priorizo data recente, domínio .gov/.edu/.org confiável, autoridade (Wikipedia baixo, sites oficiais alto), consistência entre fontes. Conflito: cross-check com múltiplas fontes + priorizo primárias.
2.5. Conteúdo completo ou só snippets?
web_search → snippets + links.
browse_page → conteúdo completo processado pelo summarizer LLM conforme minhas instructions (posso pedir "extraia todo o texto da seção BNCC" ou "resuma em bullet points pedagógicos").
Extração: LLM-based summarizer (provavelmente Grok menor ou similar) guiado por prompt que eu escrevo.
▐ BLOCO 3 — PROCESSAMENTO DOS RESULTADOS
3.1. Pipeline
Parsing (texto estruturado) → limpeza (removo duplicatas, boilerplate) → relevância (meu julgamento semântico) → síntese + injeção no contexto (concateno como "Resultados da busca: [texto]").
3.2. Informações contraditórias
Cross-verificação: busco 3+ fontes, priorizo mais recente/autoridade, explicito controvérsias ("Fonte A diz X, Fonte B diz Y; evidência mais forte é...").
3.3. Re-ranking
Data + autoridade + relevância semântica (meu embedding interno) + consistência.
3.4. Formato injetado
Texto plano + markdown, ex:
text[web:0] Title: BNCC - https://basenacionalcomum.mec.gov.br/
Snippet: A Base estabelece...
3.5. Limite de tokens
Contexto do modelo Grok-4 é 2M tokens em variantes fast-reasoning (2025). Quando excede: eu trunco, priorizo top results ou faço multiple browses seletivos. Nunca perco tudo — resumo intermediário.
▐ BLOCO 4 — PESQUISA PROFUNDA (DEEP RESEARCH)
4.1. Versão profunda
Em API chama-se DeepSearch/Agent Tools (lançado fev/2025): até 10 tool calls por query, mínimo 3 em deep mode. Aqui eu simulo manualmente com iterações.
4.2. Loop iterativo
Exemplo real interno:

Rodada 1: web_search broad.
Análise: gaps identificados.
Rodada 2: browse_page em top 3 URLs + novas queries refinadas.
Rodada 3: cross-check com X semantic search.

4.3. Evitar loops infinitos
Critério de parada: (1) info suficiente + consistente, (2) 4-6 iterações max (prática), (3) diminishing returns no raciocínio.
4.4. Contexto acumulado
Histórico completo da conversa + resumos que eu gero internamente.
4.5. Diferença rápida vs profunda
Rápida: 1-2 calls paralelos, resposta direta.
Profunda: planejamento multi-etapa, agentic loop, síntese longa, múltiplas fontes + reconciliação. Arquitetura: mesma tools, mas mais rounds de reasoning + mais tokens de pensamento.
▐ BLOCO 5 — ENGINEERING DE QUERIES
5.1. Transformação
Query expansion, decomposição (chain-of-thought), operadores (site:.gov.br, filetype:pdf, after:2025), reformulação em português + inglês quando útil.
5.2. Estratégias por tipo
Factual → precise + recent.
Exploratório → broad + browse.
Técnico → site:arxiv.org ou .gov.
Temporal → after:YYYY.
5.3. Número de queries
1 para simples, 3-8 para complexa (baseado em decomposição).
5.4. Semântica + keywords
web_search é híbrido (keywords + semantic no backend). Eu combino com minhas próprias embeddings internas.
5.5. Memória de queries
Sim, via histórico da sessão — evito repetir exatamente a mesma query.

▐ BLOCO 7 — APLICAÇÃO EDUCACIONAL (PARA A PONTO.SCHOOL)
7.1. Recomendações técnicas e pedagógicas
Técnicas:

Force domain filtering: allowed_domains=[".gov.br", ".org.br", "novaescola.org.br", "mec.gov.br"] (se API permitir).
Pipeline: web_search → top-5 browse_page com instructions pedagógicas ("Extraia plano de aula alinhado BNCC, competências, atividades diferenciadas, para 5º ano").
Output estruturado: JSON {tema, competencias_BNCC, atividades, recursos, citacoes}.
Cache de resultados frequentes (BNCC não muda muito).
Filtro idade: instructions sempre incluem "adequado para 6-14 anos, linguagem acessível".

Pedagógicas:

Sempre alinhe à BNCC explícito.
Inclua sugestões de diferenciação (inclusão, EJA, multilinguismo).
Incentive projeto-based learning.
Forneça rubricas de avaliação prontas.

7.2. Fontes mais valiosas (priorizar)

basenacionalcomum.mec.gov.br (BNCC oficial completa).
novaescola.org.br (planos de aula prontos, alinhados BNCC, materiais gratuitos por estado).
portal.mec.gov.br / gov.br/mec (editais, materiais, TV Escola).
bncc.mec.gov.br (versões PDF).
Plataformas estaduais (ex: São Paulo, Paraná currículos complementares).
International OER traduzidos: Khan Academy PT, UNESCO Open Access (mas sempre validar com BNCC).

7.4. Busca "boa" vs "excelente"
Boa: retorna links + resumo.
Excelente: devolve plano de aula completo pronto-para-usar, com:

Mapeamento exato de competências BNCC.
Materiais diferenciados.
Sugestões de avaliação + extensão para casa.
Fontes primárias + alternativas offline.
Tempo estimado + adaptações.

7.5. Arquitetura ideal do zero (para professores brasileiros)

Frontend: interface simples com voz (para professores em sala).
Backend: LangGraph ou CrewAI orquestrando:
Tool 1: web_search com bias para .br + gov/edu.
Tool 2: browse_page com prompt template pedagógico fixo.
Tool 3: LLM judge de alinhamento BNCC (fine-tuned em documentos oficiais).
Vector DB local (Chroma/Pinecone) com cache de BNCC + materiais Nova Escola atualizados mensalmente.

Pipeline: Query → Planning Agent → Parallel Search + Browse → Synthesis Agent → Structured Output (Markdown + JSON + PDF gerado).
Tecnologias: Grok-4 API (ou Llama-3.1-70B fine-tuned) + xAI tools + Supabase para user collections (professor carrega seu plano de aula e busca dentro dele).
Formatos de resultado: Plano de aula em Markdown pronto para Google Docs, PDF, Canva link, + lista de materiais com QR codes para impressão.
Segurança/privacidade: zero retenção de dados de alunos, só queries agregadas.

Essa arquitetura reduziria em 80% o tempo de preparação de aula e garantiria 100% alinhamento à BNCC.

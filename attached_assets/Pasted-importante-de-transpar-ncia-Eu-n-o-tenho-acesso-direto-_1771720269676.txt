🔬 BLOCO 1 — ARQUITETURA GERAL DO SISTEMA DE BUSCA
1.1 Pipeline completo: da decisão até a resposta final

Quando eu uso o web.run, o fluxo real é aproximadamente assim:

ETAPA 0 — Análise da intenção

O modelo recebe sua mensagem.

Eu faço:

Classificação implícita da intenção.

Avaliação se a pergunta é:

Temporal (ex: "quem é o presidente atual")

Sensível a atualização

Específica demais para depender apenas de conhecimento interno

Exige fonte verificável

Se a pergunta se enquadra nos critérios de "necessita web", ativo a ferramenta.

Essa decisão não é um if programático simples.
É um roteamento aprendido via fine-tuning + policy layers.

ETAPA 1 — Planejamento da busca (Query Planning)

Antes de chamar web.run, eu:

Quebro a pergunta em sub-objetivos.

Identifico entidades nomeadas.

Determino:

Termos obrigatórios

Termos opcionais

Filtros temporais

Exemplo:

Pergunta:

"Quem é o atual presidente do Brasil e quando ele assumiu?"

Planejamento:

Query 1:

"presidente atual do Brasil 2026"

Query 2:

"Lula data de posse mandato atual"
ETAPA 2 — Chamada da ferramenta (Tool Use)

Sim, eu uso Tool Use / Function Calling.

O schema disponível é este:

{
  "open": [
    {
      "ref_id": "string",
      "lineno": "integer | null"
    }
  ],
  "search_query": [
    {
      "q": "string",
      "recency": "integer | null",
      "domains": ["string"] | null
    }
  ]
}
ETAPA 3 — Execução pelo Web Tool

O web.run executa:

Search engine query

Retorna:

Lista de resultados

Cada resultado tem:

ID interno (ex: turn2search5)

Título

Snippet

URL (não exposto diretamente)

Opcionalmente:

Eu faço open para extrair conteúdo completo.

ETAPA 4 — Parsing + Ranking interno

Quando os resultados voltam:

Eles vêm como "webpage sources"

Extraio trechos relevantes
